# Соответствие номер карты ↔ внутренний номер (UID)

## Зависимость

- **Номер карты** (card_number) — длинный номер с карты доступа (ПИК и др.), например `01193315368729728`.
- **Внутренний номер** (internal_uid, UID) — короткий внутренний идентификатор, например `09820541`.

Связь задаётся **таблицей соответствия**, а не формулой: одному номеру карты соответствует ровно один UID.

## «Формула»

Алгебраической формулы между номером карты и UID нет. Используется справочник:

- **UID по карте:** `internal_uid = lookup(card_number)` — поиск в таблице `card_uid_mapping`.
- **Карта по UID:** `card_number = lookup_reverse(internal_uid)` — поиск по полю `internal_uid` в той же таблице.

## Данные

Список пар «номер карты → UID» импортируется из отчёта (формат строки: `card_number → internal_uid`).

Пример строки из отчёта ПИК: `01193315368729728 → 09820541`.

## База данных

- Таблица **card_uid_mapping**: `card_number` (PK), `internal_uid`. В ней хранится полный справочник.
- В таблице **users** добавлено поле **internal_uid**. Если у пользователя указан UID, по номеру карты можно однозначно найти пользователя: карта → UID → пользователь с этим `internal_uid`.

## Команды

```bash
# Создать таблицы и поле internal_uid у пользователей
npm run db:migrate-card-uid

# Загрузить соответствия из встроенного отчёта PIK454
npm run db:seed-card-uid

# Загрузить из своего файла (путь относительно проекта или абсолютный)
node server/db/seed-card-uid.js /path/to/PIK454_report.txt
```

## API

- `GET /api/card-uid/by-card/:cardNumber` — по номеру карты вернуть `internal_uid`.
- `GET /api/card-uid/by-uid/:uid` — по UID вернуть номер карты.
- `GET /api/card-uid/user-by-card/:cardNumber` — по номеру карты вернуть пользователя (если у него заполнен `internal_uid`).

Требуется авторизация.

## Алгоритм контроллера ПИК и заполнение Excel

Контроллер выдаёт UID из своей базы (при регистрации/сканировании карты), а не по формуле от номера карты. По 51+ парам (card, uid) однозначной формулы (битовое извлечение, mod, подстрока и т.п.) восстановить не удалось — значит, используется таблица или порядок регистрации в контроллере.

### Режимы заполнения

1. **Только известные (по умолчанию)**  
   Подставляются UID только для карт из эталона (первые N строк с UID в файле + пары из `data/pik-extra-reference.json`). Остальные ячейки UID остаются пустыми — такие карты нужно отсканировать на контроллере и добавить в `pik-extra-reference.json`.

2. **По порядку строк (FILL_BY_ROW=1)**  
   Если контроллер присваивает UID по порядку регистрации (первая карта — первый UID, вторая — второй и т.д.), то неизвестные строки заполняются последовательно от последней известной. Последняя известная строка определяется по файлу и по парам из `pik-extra-reference.json` (например, строка 51 с UID 09830317). Строки 52, 53, … получают 09830318, 09830319, … Рекомендуется, если порядок строк в Excel совпадает с порядком регистрации карт в контроллере.

3. **С интерполяцией (FILL_UNKNOWN=1)**  
   Для карт вне эталона UID подставляются по интерполяции между ближайшими эталонными парами (по номеру карты), с разрешением коллизий. Таблица получается полностью заполненной, но значения для неотсканированных карт — приближённые.

Во всех режимах номер карты для поиска в эталоне нормализуется (только цифры, дополнение до 17 знаков), чтобы совпадение работало и когда Excel возвращает число без ведущего нуля.

```bash
# Только известные с контроллера (остальные пусто)
node scripts/fill-xlsx-uids.js "/путь/к/Копия PIK454_report.xlsx" "/путь/к/результат.xlsx"

# По порядку строк (последовательно от последней известной, напр. 51 → 09830318, 52 → 09830319…)
FILL_BY_ROW=1 node scripts/fill-xlsx-uids.js "/путь/к/Копия PIK454_report.xlsx" "/путь/к/результат.xlsx"

# Заполнить и неизвестные по интерполяции по номеру карты
FILL_UNKNOWN=1 node scripts/fill-xlsx-uids.js "/путь/к/Копия PIK454_report.xlsx" "/путь/к/результат.xlsx"
```

Дополнительные пары (card, uid) с контроллера добавляйте в **data/pik-extra-reference.json** в формате: `{ "card": "01246255135620224", "uid": "09830317" }`.
